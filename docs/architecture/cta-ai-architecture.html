Example: "What was Q1 revenue by region?"

Layer 1-2 Alice (Finance) opens Chrome, already signed in via Okta SSO Cognito refreshes her JWT. Amplify serves the React SPA. She sees the chat interface.
↓
Layer 2 Alice types: "What was Q1 revenue by region?" and hits Enter React sends a GraphQL mutation to AppSync. The request includes her JWT token with Finance group.
↓
Layer 3 AppSync validates JWT, invokes queryHandler Lambda queryHandler orchestrates: ragRetriever searches Aurora, sqlGenerator calls Claude, sqlExecutor runs the SQL.
↓
Layer 4-5 RAG retrieval: Titan embeds the question, Aurora finds similar definitions Aurora returns: "Revenue = dues + sponsorships" and "fct_revenue joined to dim_region". Claude receives context + the question.
↓
Layer 4 Claude generates SQL + natural language explanation SELECT r.region, SUM(f.dues + f.sponsorships) FROM revenue_metric_vw... + "This query sums dues and sponsorships for Q1 2026 by region."
↓
Layer 3 sqlExecutor validates (no DROP/DELETE/INSERT), runs in Sandbox first The Sandbox warehouse resumes from auto-suspend (2-5s if idle), executes the SQL. Query returns in 2.3 seconds. Success.
↓
Layer 6 Sandbox passed. Same SQL runs in Production warehouse. Snowflake RLS automatically appends WHERE org_id = 'FINANCE'. Alice sees 4 Finance regions. Bob (Membership) would see different rows.
↓
Layer 3 queryHandler formats the response, sends back via AppSync JSON payload: summary text, chart data (Recharts bar chart), table data (first 100 rows), generated SQL, confidence score.
↓
Layer 2 React renders: summary card + expandable chart + data table Alice sees: "Q1 2026 revenue was $2.4M, up 12% from Q4 2025." She clicks "Show Chart" and sees a bar chart. She clicks "Show SQL" to verify.
↓
Layer 7 Background (async): auditLogger writes to S3 + CloudWatch S3: JSON file with Alice's question, generated SQL, result hash, 2.3s latency, 0.03 credits. CloudWatch: latency metric incremented.
↓
Layer 2 Alice gives thumbs-down: "Revenue should include event income too" EventBridge fires an async event. Alice gets instant confirmation ("Feedback recorded"). A nightly Lambda will analyze patterns and propose a YAML change for review.
Total end-to-end time: ~4-6 seconds  |  Total cost: ~$0.05 per query
+
Teach Mode: How the AI Learns from Feedback

When users give thumbs-down feedback, the system analyzes patterns and proposes semantic model updates. Here's the full loop.

Layer 2
Alice clicks thumbs-down and types: "Revenue should include event income too"
React captures the feedback and sends it via GraphQL. Alice gets instant confirmation: "Feedback recorded." No waiting.
↓
Layer 3
EventBridge queues the feedback event asynchronously
The event is stored in Aurora's feedback table with status "pending_analysis". Alice's UI response is immediate because this happens in the background.
↓
Layer 3
Nightly batch Lambda (2 AM UTC): analyzeFeedbackBatch runs
The Lambda queries all "pending_analysis" feedback from the last 24 hours. It groups similar corrections using text similarity. If 3+ users mentioned "event income" and "revenue", it forms a pattern cluster.
↓
Layer 4
Claude analyzes the pattern and proposes a YAML change
Prompt: "Current definition: Revenue = dues + sponsorships. User corrections: [3 examples]. Propose a YAML change." Claude returns the updated metric formula and new synonyms.
↓
Layer 8
GitHub PR is created by cta-ai-bot
Title: "Feedback: Update revenue definition (3 users)". Body includes: AI analysis summary, proposed YAML diff, affected queries. Assigned to the data steward for review.
↓
Layer 8
Data steward reviews and approves the PR
Human checks: Does event_income column exist in fct_revenue? Will this break existing reports? Steward merges PR if valid, or comments with corrections.
↓
Layer 8
CI/CD auto-deploys: dbt test, build views, regenerate embeddings
GitHub Actions runs: dbt tests pass, revenue_metric_vw is updated with new formula, Titan generates new embeddings, Aurora pgvector is updated. Deploys in ~5 minutes.
↓
Layer 2
Alice asks again: "What was Q1 revenue?" -- now includes events
Aurora retrieves the updated definition. Claude generates SQL with event_income. Alice sees $2.7M (up from $2.4M). Feedback loop closed.
Why this works
  • Non-blocking: Alice never waits; feedback is queued instantly
  • Batched: Nightly analysis finds patterns (3+ users) before acting
  • AI-assisted: Claude proposes the change; human decides
  • Version-controlled: GitHub PR = full audit trail of why definition changed
  • Tested: dbt validates before production deployment
